# GameAssistant Web 平台开发循环文档

> **项目名称**: GameAssistant Model Training Platform  
> **需求文档**: `docs/web-platform-requirements_2026-06-22.md`  
> **触发方式**: 固定间隔（每 10 分钟）  
> **当前状态**: 初始化（第 1 轮尚未开始）

---

## 一、文档用途

本文档是整个 Loop 工程的执行蓝图。每一次触发时，新建的任务 agent（开发）和评估 agent 都必须完整阅读本文档，以理解：

1. 当前项目要做什么（需求文档）
2. 本次 loop 的任务范围（由任务建议.md 指定）
3. 本次 loop 的产出要求（报告文档格式、文件夹结构）
4. 循环如何推进（下次 loop 读什么）

---

## 二、需求文档

开发的目标系统是一个 **本地 Web 管理平台**，技术栈：

- **前端**: Vue 3.4+ (Composition API) + TypeScript + Vite + Ant Design Vue 4.x + Pinia
- **后端**: FastAPI + Python 3.10+
- **数据库**: MySQL 8.0+（Flyway 迁移管理）
- **实时通信**: Server-Sent Events (SSE)

完整需求见：`docs/web-platform-requirements_2026-06-22.md`

## 三、循环执行流程

每次 Loop 触发（即每 10 分钟），按以下顺序执行：

```
┌─────────────────────────────────────────────────────────────┐
│                    Loop 触发（每 10 分钟）                     │
└────────────────────────────┬────────────────────────────────┘
                             ▼
┌──────────────────────────────────────────────────────────────┐
│ 步骤 1：新建【开发 Agent】                                      │
│  1. 读取本文档（loop.md）                                       │
│  2. 如果不是第 1 轮循环 → 读取上一次循环的 "任务建议.md"             │
│     （第 1 轮不读，因为还没有上一次建议）                           │
│  3. 读取需求文档（web-platform-requirements_2026-06-22.md）       │
│  4. 根据任务建议.md 确定本次开发范围                               │
│  5. 执行开发实现（代码编写、配置、迁移脚本等）                       │
│  6. 在 docs/loop/loop-XXX/ 下创建 "修改与测试报告.md"（初稿）       │
└────────────────────────────┬───────────────────────────────────┘
                             ▼
┌──────────────────────────────────────────────────────────────┐
│ 步骤 2：执行【功能测试】                                          │
│  1. 启动后端服务（验证 API 可用）                                  │
│  2. 启动前端服务（验证页面可加载）                                  │
│  3. 逐项测试本次实现的功能点                                       │
│  4. 记录测试结果到 "修改与测试报告.md" 中                            │
│  5. 如测试失败 → 在报告中标注问题 → 下一轮继续修复                   │
└────────────────────────────┬───────────────────────────────────┘
                             ▼
┌──────────────────────────────────────────────────────────────┐
│ 步骤 3：新建【评估 AGENT】                                       │
│  1. 读取本文档                                                 │
│  2. 读取需求文档                                                │
│  3. 读取本次循环的 "修改与测试报告.md"                              │
│  4. 读取本次循环已完成的代码                                       │
│  5. 评估本次实现是否完整、是否符合需求                              │
│  6. 对照需求文档和下一阶段目标，提出下一次 loop 的具体开发建议         │
│  7. 将评估结果写入 "任务建议.md"（放在本次 loop-XXX 文件夹下）        │
└────────────────────────────┬───────────────────────────────────┘
                             ▼
┌──────────────────────────────────────────────────────────────┐
│ 循环完成，等待下一次触发（10 分钟后）                               │
└──────────────────────────────────────────────────────────────┘
```

## 五、任务建议.md 查找规则

- **第 1 轮循环**：开发 agent **不读**任务建议.md（因为没有上一次）
- **第 2 轮循环**：开发 agent 读取 `docs/loop/loop-001/任务建议.md`
- **第 3 轮循环**：开发 agent 读取 `docs/loop/loop-002/任务建议.md`
- 以此类推，总是指向上一次循环的文件夹

如果某次循环未生成 `任务建议.md`（例如 agent 执行中断），下一次开发 agent 跳过读取即可，无需等待。

## 六、循环推进规则

### 6.1 何时认定"完成"

当评估 agent 确认以下全部满足时，本次循环才算完成：

1. 本次实现的功能符合需求文档中的描述
2. 功能测试通过（或已记录的失败项在报告中标注）
3. `修改与测试报告.md` 和 `任务建议.md` 均已生成
4. 代码已提交到当前工作分支（推荐每个循环做一次 commit）

### 6.2 循环终止条件

满足以下任一条件时，循环可以终止：

1. 需求文档中的所有 P0 功能全部实现并测试通过
2. 用户主动要求停止
3. 连续 3 轮无实质性功能进展（由评估 agent 判断）

## 七、开发 Agent 行为准则

1. **每次开发前先读 loop.md**，确认当前是第几轮、上次任务建议是什么
2. **以需求文档为准绳**：所有实现必须严格对照 `web-platform-requirements_2026-06-22.md`
3. **单次循环聚焦**：每轮只做一个小而完整的单元（如一个模块的后端 API + 前端页面），避免贪多嚼不烂
4. **测试必须执行**：没有测试结果的报告等于无效报告
5. **遗留问题要显式记录**：不要假装没看到问题，忠实记录才能在下轮修复
6. **不做需求外扩展**：除非任务建议.md 明确提到，否则不主动增加功能

---

## 八、评估 Agent 行为准则

1. **诚实评估**：不要为了"好看"而夸大完成度，真实问题必须指出
2. **具体可操作**：每条建议必须是开发 agent 能直接执行的任务，不能是模糊的方向性描述
3. **优先建议 P0**：把最重要的功能改进建议放在最前面
4. **关注衔接**：评估时要考虑本次与上次的衔接，确保代码能合到一起
