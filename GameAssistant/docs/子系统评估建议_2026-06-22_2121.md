## 二、第二轮新发现问题

经过深度分析，更新后的需求文档在以下方面发现新问题。

### 2.1 数据库设计新问题

#### D-N1：annotation_projects.class_ids JSON 悬空引用风险 ⚠️ 中优先级

**问题描述**：`annotation_projects` 表中 `class_ids` 字段为 JSON 数组（UUID 列表），但 `classes` 表删除记录后，JSON 数组中的 UUID 会变成悬空引用。

**影响**：标注项目关联的类别被删除后，数据库层面不会自动级联清理，项目仍显示已删除的类别 ID。

**修复建议**：在应用层（CRUD 层）增加类别删除时的级联逻辑：当 `classes` 记录被删除时，遍历所有 `annotation_projects`，从 `class_ids` JSON 数组中移除该 UUID。

```python
# crud/classes.py - 删除类别时的级联清理
async def delete_class(db: AsyncSession, class_id: str) -> None:
    # 1. 从所有 annotation_projects.class_ids 中移除该类别
    projects = await db.execute(
        select(AnnotationProject).where(AnnotationProject.class_ids.contains(class_id))
    )
    for project in projects.scalars():
        new_ids = [cid for cid in project.class_ids if cid != class_id]
        project.class_ids = new_ids
    # 2. 删除类别记录
    await db.execute(delete(Class).where(Class.id == class_id))
```

#### D-N2：models.class_ids 同样存在悬空引用风险 ⚠️ 中优先级

**问题描述**：与 D-N1 相同，`models.class_ids` JSON 字段在 `classes` 表记录删除后也会产生悬空 UUID。

**影响**：模型元信息中显示已删除的类别，导致模型详情页显示异常。

**修复建议**：同上，在类别删除时同步清理 `models.class_ids` JSON 数组。

#### D-N3：Flyway V2 迁移脚本中 FK 外键约束可能缺失 ⚠️ 低优先级

**问题描述**：文档 V2 迁移脚本路径为 `V2__add_annotation_project_images.sql`，但该脚本仅创建 `annotation_project_images` 表，不包含 `annotation_projects` 和 `images` 表本身。`annotation_projects` 表在 V3 中才创建。

**影响**：如果 V2 迁移先于 V3 执行，FK 约束无法正确添加（目标表不存在）。

**修复建议**：
- **方案 A**：将 `annotation_project_images` 的创建合并到 V3（与 `annotation_projects` 同一迁移文件）
- **方案 B**：V2 迁移脚本中仅记录迁移说明，实际建表延后到 V3
- **方案 C**：确保 Flyway 依赖顺序正确（V3 必须在 V2 之后执行，且 V2 不应创建依赖 V3 存在的表）

> 注：V1 包含 `users, images, source_videos, classes`，V3 包含 `annotations, annotation_projects, reviews`，V2 独立创建 `annotation_project_images` 在 V1 和 V3 之间可能存在 FK 依赖问题。**建议将 V2 内容合并到 V3**。

#### D-N4：users.is_active 命名不一致问题仍然存在 ⚠️ 低优先级

**问题描述**：第一轮已指出 `users` 表使用 `is_active` 而其他表使用 `is_deleted`，但更新文档中仍未统一。

**当前状态**：
- `users`: `is_active TINYINT(1) NOT NULL DEFAULT 1`（激活状态）
- `images`, `datasets`, `models`, `templates`: `is_deleted TINYINT(1)`（软删除）

**影响**：用户管理的逻辑与其他实体的软删除逻辑不统一，代码中需要分别处理两种语义。

**修复建议**：将 `users.is_active` 视为用户禁用（对应 `is_deleted=1` 表示用户已删除/禁用），统一使用 `is_active` 语义或统一使用 `is_deleted` 语义。建议保持 `is_active`（语义更清晰），在代码规范中说明这是唯一例外。

#### D-N5：training_jobs.log_summary 与 training_logs 表数据冗余 ⚠️ 低优先级

**问题描述**：`training_jobs.log_summary` 为 JSON 字段，`training_logs` 表也存储 epoch 级别的日志，两者存在数据冗余。

**影响**：冗余数据增加存储成本，两处数据需要保持同步。

**修复建议**：移除 `training_jobs.log_summary` JSON 字段，仅保留 `training_logs` 表作为训练日志的唯一数据源。在需要获取日志摘要时，通过 `SELECT epoch, map50, map50_95 FROM training_logs WHERE training_job_id = ? ORDER BY epoch` 查询。

### 2.2 架构设计新问题

#### A-N1：wrappers 层缺少错误处理和重试机制 ⚠️ 中优先级

**问题描述**：文档 3.5 表格中描述了 wrappers 的调用接口，但没有说明错误处理策略。对于 ADB 截图和训练任务来说，网络波动和进程异常是常态。

**建议补充**：

| Wrapper | 错误场景 | 建议处理策略 |
|---|---|---|
| ADBWrapper | adb 连接断开、截图超时 | 3 次重试 + 指数退避 + 抛出自定义异常 |
| TrainerWrapper | 进程崩溃、日志解析失败 | 捕获 exit code，记录 error_message |
| DetectorWrapper | GPU OOM、模型加载失败 | 单例模式管理 detector 生命周期 |
| TemplateWrapper | 模板文件不存在 | 启动时预加载，文件缺失抛出异常 |

#### A-N2：docker-compose 中 Redis 的必要性存疑 ⚠️ 低优先级

**问题描述**：文档提到 Redis 为"备选，第一阶段可省略"，但 docker-compose.yml 中包含了 Redis 服务。文档 3.1 技术栈表中说明 Redis 用于"SSE 状态存储 / Celery 消息队列"。

**分析**：
- **第一阶段**：仅使用 subprocess 管理训练，无 SSE 状态存储需求（训练任务在内存中），Redis 在第一阶段确实用不上
- **预埋价值**：如果预埋了 Redis，后续接入 Celery 任务队列或 SSE 连接池会更容易
- **成本**：Docker 启动一个闲置的 Redis 容器几乎没有资源消耗

**结论**：Redis 预埋是合理的架构演进预留，**不是问题**。但建议在 docker-compose.yml 中添加注释说明 Redis 的用途和启用条件，避免后续维护者困惑。

#### A-N3：前端 stores 和 api 目录缺少路由懒加载设计 ⚠️ 低优先级

**问题描述**：文档中的前端目录结构没有说明路由懒加载（lazy loading）策略。对于内部工具来说，首屏加载速度影响用户体验。

**建议**：在 `router/index.ts` 中使用 Vue Router 的动态导入：

```typescript
const routes = [
  {
    path: '/annotation',
    component: () => import('@/views/Annotation.vue'), // 懒加载
  },
]
```

对于标注画布等重型组件，懒加载可以显著减少首屏 bundle 大小。

#### A-N4：缺少前端 API 类型自动生成方案 ⚠️ 低优先级

**问题描述**：文档 3.3.6 提到"配合 openapi2ts 或 orval 自动生成 TypeScript 类型"，但没有具体说明选型。

**建议**：推荐使用 **orval**，因为它：
- 支持 FastAPI 的 OpenAPI 3.0 schema 直接生成 TypeScript
- 可以生成 Zod 验证 schema
- 支持 React Query / SWR / Axios 集成
- 社区活跃，Vue 生态兼容性好

## 四、docker-compose 架构评估

### 4.1 Redis 引入合理性分析

**第一阶段定位**：文档 3.1 明确说明 Redis 为"备选，第一阶段可省略"，用于"SSE 状态存储 / Celery 消息队列"。

**评估**：

| 维度 | 分析 | 结论 |
|---|---|---|
| 第一阶段必要性 | 当前使用 subprocess 管理训练，SSE 状态在内存中 | Redis 在第一阶段确实用不上 |
| 预埋价值 | 后续接入 Celery 任务队列 + SSE 连接池 + Session 缓存 | ✅ 有价值 |
| 资源成本 | Docker 闲置 Redis 几乎无资源消耗 | ✅ 可忽略 |
| 复杂度增加 | Docker Compose 中多一个服务 | ✅ 极小增量 |
| 文档说明 | 需在 docker-compose.yml 中注释说明用途 | ⚠️ 建议补充 |

**结论**：Redis 预埋是**合理的架构演进预留**，不是问题。但建议在 docker-compose.yml 中添加注释：

```yaml
# Redis (optional): Enable for distributed training queue (Celery) or SSE connection pooling
# Comment out services.redis if not needed in Phase 1
```

### 4.2 docker-compose 服务清单

```
services:
  mysql:8.0          ← 核心数据存储，必需
  redis:7-alpine      ← 预留，可选
  backend:            ← FastAPI 应用，必需
  frontend:           ← Vue3 构建产物，必需（可选：直接 nginx 服务）
```

**评估结论**：服务清单合理，MySQL 和 Backend/Frontend 为主服务，Redis 预留。
